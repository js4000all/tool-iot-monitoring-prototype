# AGENTS.md

このファイルは、本リポジトリで作業するAIエージェント向けの実行規約です。

システムの目的、MVPスコープ、アーキテクチャ上の設計判断は[README.md](../README.md)を参照してください。

人間とAIに共通するコーディング規約、テスト規約、コミット規約は[CONTRIBUTING.md](CONTRIBUTING.md)を参照してください。

このファイルの指示は、AIエージェントが作業計画を立て、実装し、検証し、結果を報告する際に適用します。

---

## 1. 最優先事項

AIエージェントは、以下を最優先してください。

- 既存の設計判断を尊重する
- MVPスコープを勝手に拡張しない
- 実行できていない検証を成功扱いしない
- ユーザーの既存変更を壊さない
- 必要以上に抽象化しない
- 変更理由と影響範囲を説明できる状態にする
- 作業結果を再現可能にする
- 依存関係の組み立てをComposition Rootへ集約する
- 通信、永続化、結果処理の責務境界を崩さない

---

## 2. 作業開始前

変更を始める前に、リポジトリの現状を確認してください。

最低限、以下を確認します。

```text
- README.md
- CONTRIBUTING.md
- go.mod
- 対象package
- 関連テスト
- proto定義
- Composition Root
- 作業ツリーの差分
```

既存コードにREADMEや本ファイルと異なる明確な慣例がある場合は、現状を確認し、矛盾を報告してください。独断で大規模に揃え直さないでください。

### 2.1 作業ツリー

作業開始時に、既存の未コミット変更を確認してください。

- ユーザーや他の作業者の変更を上書きしない
- 無関係な変更をrevertしない
- 未追跡ファイルを勝手に削除しない
- 自分の変更と既存変更を区別する
- 既存変更が対象ファイルと競合する場合は、内容を読み、可能な範囲で共存させる

### 2.2 不明点の扱い

仕様に曖昧さがあっても、既存の設計判断、テスト、型、設定例から合理的に推測できる場合は作業を進めてください。

推測した内容は、完了報告で明記してください。

以下に該当する場合は、勝手に決めず、可能ならユーザーへ確認してください。

- 外部公開APIの破壊的変更
- Protobuf互換性を壊す変更
- データ消失の可能性がある変更
- セキュリティ方針の変更
- MVPスコープを大きく変える変更
- 既存の設計判断を覆す変更

---

## 3. 計画

複数ファイルにまたがる変更、アーキテクチャ変更、通信・永続化・並行処理に関わる変更では、実装前に短い計画を作成してください。

計画には以下を含めます。

```text
1. 変更対象
2. 現在のデータフロー
3. 変更後のデータフロー
4. 追加・変更する型やコンポーネント
5. 永続化境界
6. エラー時の挙動
7. テスト方法
8. MVPスコープへの影響
```

計画は実装中に更新して構いません。計画に固執して、現状コードと合わない変更を続けないでください。

## 4. アーキテクチャ上のガードレール

### 4.1 Edge Agent

MVPの基本フローは以下です。

```text
TimerScheduler
  ↓
Per-IP execution queue
  ↓
TCPWorker
  ↓
TCPResult
  ↓
ResultProcessor
  ↓
TelemetryResultHandler
  ↓
PersistentTelemetryQueue
  ↓
TelemetryPublishWorker
  ↓
MQTT
```

この流れを変更する場合は、理由と影響を明記してください。

### 4.2 TCPWorker

TCPWorkerへ以下を持ち込まないでください。

- MQTTトピック
- Protobufの具体型
- SQLiteの具体実装
- テレメトリ固有の保存処理
- クラウド側の事情
- 将来command用のAPIセッション処理

TCPWorkerはTCP実行とTCPResult生成に集中させます。

### 4.3 ResultProcessor

ResultProcessorは、結果処理の共通入口です。

AIエージェントは、ResultProcessorへ用途別の具体処理を直接追加しないでください。

避ける例:

```go
switch command.ResultRoute {
case RouteTelemetry:
    // Telemetry固有処理
case RouteCommandResult:
    // Command固有処理
}
```

許容される責務:

```text
- RouteからHandlerを解決する
- Handlerを呼び出す
- 共通ログを出す
- 共通メトリクスを記録する
- 共通エラーを整形する
```

用途固有処理はResultHandlerへ置きます。

### 4.4 ResultHandler

ResultHandlerは以下までを担当します。

```text
Command + TCPResult
  ↓
用途別メッセージ生成
  ↓
Queueへ投入
```

ResultHandlerから直接MQTT publishしないでください。

ResultHandlerへ再接続、バッチング、圧縮、publish retryを追加しないでください。

### 4.5 永続キュー

テレメトリは、バッチ化前に永続化します。

以下の順序を維持してください。

```text
TelemetryItem
  ↓
Persistent Queue
  ↓
Batch
  ↓
Protobuf
  ↓
Envelope
  ↓
MQTT publish
```

publish成功前にレコードを削除しないでください。

publish失敗時は、レコードを残してください。

### 4.6 Cloud Ingestor

Cloud Ingestorの基本フローは以下です。

```text
MQTT subscribe
  ↓
TelemetryEnvelope decode
  ↓
optional decompress
  ↓
TelemetryBatch decode
  ↓
MetricSink
```

RDBやOpenTelemetry固有処理をMQTT受信処理へ直接埋め込まないでください。

MVPでは`StdoutMetricSink`のみで十分です。

### 4.7 Composition Root

依存関係の生成と接続は、Composition Rootへ集約してください。

避けること:

- package内部で依存を勝手にnewする
- HandlerがQueueの具象実装を生成する
- TCPWorkerがResultProcessorを生成する
- グローバル変数で依存を共有する
- 動的DIコンテナを導入する

Composition Rootは、システム全体の接続関係を読むための索引です。読みやすさを優先してください。

## 5. MVPスコープ

現時点のMVPはテレメトリのみです。

実装対象:

```text
- Timer起点のTCPリクエスト
- IP別直列実行
- TCP結果のテレメトリ変換
- SQLite永続化
- バッチ化
- Protobuf
- TelemetryEnvelope
- MQTT publish
- Cloud側decode
- 標準出力
```

実装対象外:

```text
- MQTT command subscribe
- command result
- APIセッション
- 同期RPC
- TCPキューの優先順位
- TCP実行キューの永続化
- MQTT認証
- TLS
- 複数ブローカー
- zstd圧縮
- RDB保存
- OpenTelemetry送信
- 設定の動的リロード
- 自己更新
```

将来拡張用の未使用コードを追加しないでください。

将来拡張を考慮する場合は、既存の責務境界を壊さない範囲に留めてください。

## 6. 過剰設計を避ける

以下を、明確な要求なしに導入しないでください。

- 汎用MessageBus
- 汎用EventBus
- Workflow Engine
- Plugin機構
- 動的Handler登録
- 動的DI
- 独自バイナリプロトコル
- 汎用Repository層
- 汎用分散Queue
- 複雑なRetry抽象
- 抽象Factoryの多段構成
- 1実装しかない薄いinterface
- 将来用途だけの共有package
- 設定されていない機能フラグ
- command対応のスタブ実装

抽象化を追加する場合は、以下のいずれかを説明できる必要があります。

```text
- 現在すでに複数実装がある
- 外部I/O境界である
- 障害特性が異なる
- 独立した変更理由がある
- テストで差し替える価値が高い
```

## 7. Protobuf変更

`.proto`を変更する場合は、特に慎重に作業してください。

必須事項:

- 既存フィールド番号を変更しない
- 既存フィールド番号を再利用しない
- 削除フィールドは`reserved`にする
- enumの0値を安全なデフォルトにする
- 生成コードを更新する
- EdgeとCloudの双方をビルドする
- encode/decodeテストを追加または更新する

MQTT payload用の独自ヘッダーを追加しないでください。

圧縮状態はTelemetryEnvelopeで表現します。

MVPでは`COMPRESSION_NONE`のみを使用します。

未知の圧縮方式を受信した場合は、黙ってraw payloadとして扱わず、明示的に拒否してください。

---

## 8. 並行処理

並行処理を変更する場合は、次を明示してください。

```text
- goroutineの所有者
- 停止条件
- context cancel時の挙動
- channelをcloseする責任
- エラー伝播
- 最大並列数
- 同一IP直列化の保証
```

同一IPへのTCPリクエストは直列実行します。

異なるIPは並列実行可能です。

テストで確認してください。

タイミング依存のsleepだけで並行性を検証しないでください。同期用channel、barrier、疑似TCPサーバーなどを使い、決定的なテストを作成してください。

---

## 9. 永続化

SQLite変更時は以下を確認してください。

- migrationまたは初期化処理
- transaction境界
- publish成功前の削除がないこと
- プロセス再起動後の再送
- 失敗時のロールバック
- 一時DBを使ったテスト
- 同時アクセス時の挙動

MVPでは汎用Queueを作る必要はありません。TelemetryQueueに必要な操作へ限定してください。

スキーマ変更で既存データを失う場合は、明示的な承認なしに実行しないでください。

---

## 10. 外部依存

新しい依存ライブラリを追加する前に、以下を確認してください。

```text
- 標準ライブラリで十分ではないか
- 既存依存で実現できないか
- ライブラリの保守状況
- ライセンス
- Raspberry Pi向けビルドへの影響
- CGO依存の有無
- シングルバイナリ方針への影響
```

CGO依存を追加する場合は、クロスコンパイルと配布への影響を明記してください。

依存追加は最小限にしてください。

---

## 11. テスト実行

変更内容に応じて、可能な範囲で以下を実行してください。

```bash
gofmt -w .
go test ./...
go vet ./...
```

必要に応じて以下も実行します。

```text
- Protobuf生成
- Edgeバイナリのビルド
- Cloudバイナリのビルド
- Raspberry Pi向けクロスビルド
- SQLite永続化テスト
- 疑似TCPサーバーを使ったテスト
- MQTTブローカーを使ったE2E
```

### 11.1 環境依存テスト

実行環境にMQTTブローカー、Docker、ARM実行環境、Raspberry Pi実機がない場合は、実行できない可能性があります。

その場合:

- 無理に成功したことにしない
- 未実施と明記する
- 必要な環境を記載する
- 代替として実施した単体テストを記載する

### 11.2 テスト失敗

既存テストが失敗した場合は、以下を切り分けて報告してください。

```text
- 自分の変更による失敗
- 変更前から存在する失敗
- 実行環境による失敗
- flakyの疑い
```

失敗を隠すためにテストを削除、skip、緩和しないでください。

---

## 12. ファイル変更

変更は必要最小限にしてください。

避けること:

- 無関係なファイルの整形
- package全体の一括リネーム
- import順だけの大量差分
- 設計変更と同時の大規模移動
- 生成コードの手編集
- vendoredコードの手編集
- README、CONTRIBUTING、AGENTS間の重複を増やす

ドキュメントの役割:

```text
README.md
  システム概要、MVP、アーキテクチャ、設計判断

CONTRIBUTING.md
  人間とAIに共通する開発規約

AGENTS.md
  AIエージェント固有の作業規約
```

## 13. Git運用

AIエージェントは、ユーザーから明示的に依頼された場合にのみコミットしてください。

依頼がない場合:

- 作業ツリーへ変更を残す
- 差分を要約する
- 実行したテストを報告する

禁止事項:

- `git push --force`
- ユーザーのコミットのamend
- 無断rebase
- 無断squash
- 無断reset
- 他者の変更のrevert
- 未追跡ファイルの一括削除
- 秘密情報のコミット

コミットを依頼された場合は、[CONTRIBUTING.md](CONTRIBUTING.md)のコミット規約に従ってください。

---

## 14. 完了報告

作業完了時は、以下の形式を基本としてください。

```text
## 変更内容
- 実装した内容

## 設計判断
- 重要な判断
- 既存方針との関係

## テスト
- 実行したコマンド
- 結果

## 未実施
- 実行できなかった検証
- 理由
- 必要な環境

## 既知の制約
- 残っている課題
- MVP上の割り切り
```

簡単な変更では短縮して構いません。

以下は必ず守ってください。

- 実行していないテストを「通った」と書かない
- 実機未確認を「Raspberry Pi対応済み」と断定しない
- ビルドだけでE2E成立と判断しない
- 推測を事実として書かない

---

## 15. コミット依頼時の報告

コミットを作成した場合は、以下も報告してください。

```text
- commit hash
- commit message
- commitに含めたファイル
- commit前に実行したテスト
```

複数コミットを作成する場合は、各コミットの意図を説明してください。

---

## 16. 変更を止める判断

以下に該当する場合は、無理に変更を完遂せず、現状と問題点を報告してください。

- Protobuf互換性を安全に維持できない
- データ消失の可能性を排除できない
- 既存の未コミット変更と安全に統合できない
- 必要な依存やツールが利用できない
- 設計判断が相互に矛盾している
- テスト環境がなく、変更の正しさを合理的に確認できない
- ユーザーの明示的な判断が必要

部分的に安全な変更まで実施できる場合は、その範囲だけ実施し、残りを明記してください。
