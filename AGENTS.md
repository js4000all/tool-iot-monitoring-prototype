システムの目的、MVP スコープ、アーキテクチャ上の設計判断、開発規約は以下を参照してください。

- [README.md](README.md)

- [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)
- [docs/CONTRIBUTING.md](docs/CONTRIBUTING.md)

---

## 1. 最優先事項

AI エージェントは、以下を最優先してください。

- 既存の設計判断を尊重する
- MVP スコープを勝手に拡張しない
- 実行できていない検証を成功扱いしない
- ユーザーや他の作業者の既存変更を壊さない
- 必要以上に抽象化しない
- 変更理由と影響範囲を説明できる状態にする
- 作業結果を再現可能にする
- 依存関係の組み立てを Composition Root へ集約する
- 通信、永続化、結果処理の責務境界を崩さない

---

## 2. 作業開始前

変更を始める前に、最低限以下を確認してください。

```text
- README.md
- docs/MVP.md
- docs/ARCHITECTURE.md
- docs/CONTRIBUTING.md
- go.mod
- 対象 package
- 関連テスト
- proto 定義
- Composition Root
- 作業ツリーの差分
```

既存コードにドキュメントと異なる明確な慣例がある場合は、現状を確認し、矛盾を報告してください。独断で大規模に揃え直さないでください。

### 2.1 作業ツリー

- ユーザーや他の作業者の変更を上書きしない
- 無関係な変更を revert しない
- 未追跡ファイルを勝手に削除しない
- 自分の変更と既存変更を区別する
- 既存変更が対象ファイルと競合する場合は、内容を読み、可能な範囲で共存させる

### 2.2 不明点の扱い

仕様に曖昧さがあっても、既存の設計判断、テスト、型、設定例から合理的に推測できる場合は作業を進めてください。推測した内容は完了報告で明記してください。

以下に該当する場合は、勝手に決めず、可能ならユーザーへ確認してください。

- 外部公開 API の破壊的変更
- Protobuf 互換性を壊す変更
- データ消失の可能性がある変更
- セキュリティ方針の変更
- MVP スコープを大きく変える変更
- 既存の設計判断を覆す変更

## 3. 計画

複数ファイルにまたがる変更、アーキテクチャ変更、通信・永続化・並行処理に関わる変更では、実装前に短い計画を作成してください。

計画には以下を含めます。

```text
1. 変更対象
2. 現在のデータフロー
3. 変更後のデータフロー
4. 追加・変更する型やコンポーネント
5. 永続化境界
6. エラー時の挙動
7. テスト方法
8. MVP スコープへの影響
```

計画は実装中に更新して構いません。計画に固執して、現状コードと合わない変更を続けないでください。

---

## 4. 実装時の禁止事項

- MVP に不要な汎用化をしない
- 将来の command 機能を先回りして実装しない
- ResultProcessor へ用途別の具体処理を直接追加しない
- ResultHandler から MQTT を直接操作しない
- TCPWorker からテレメトリ固有型や MQTT 詳細を参照しない
- 永続化成功と publish 成功を混同しない
- Protobuf の既存フィールド番号を再利用しない
- MQTT payload 用の独自バイナリヘッダーを作らない
- 動的 DI コンテナや Handler の実行時追加を導入しない
- 利用箇所のない interface を将来用途だけで追加しない

詳細な設計上のガードレールは [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) を参照してください。

---

## 5. 検証と報告

- 実行したコマンドと結果を明記する
- 実行できなかった検証は、理由と必要な環境を明記する
- 未実施の検証を成功扱いしない
- 変更したファイル、設計判断、影響範囲を報告する
- コミットを作成した場合は、コミットハッシュとコミットメッセージを報告する

変更完了時の報告形式は [docs/CONTRIBUTING.md](docs/CONTRIBUTING.md) の「変更完了時の報告」を参照してください。
